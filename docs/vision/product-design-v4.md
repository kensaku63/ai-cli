# ai-cli プロダクト設計: v4 ビジョンに基づく本質的フォーカス

**日付**: 2026-04-09
**著者**: make-vision.kensaku63
**背景**: v4 ビジョン「AIの知性に、人類のソフトウェア全てを繋ぐ」に基づき、何にフォーカスすべきかを設計

---

## 1. 根本的な問い: ai-cli のプロダクトは何か

**ai-cli のプロダクトはCLIツールではない。ソフトウェアの知識である。**

CLIツールは配送手段にすぎない。ai-cli が本当に提供するのは:

- `jq` が JSON パースに最適だという知識
- `ripgrep` が `grep` より高速だという知識
- `curl | jq | xargs` が強力な合成パターンだという知識
- 昨日リリースされた新しいCLIツールの存在と使い方

この知識こそが、AI の潜在能力と実効能力のギャップを埋める。

**アナロジー:**

| | Google | ai-cli |
|---|---|---|
| プロダクト | Webページの知識（インデックス） | ソフトウェアツールの知識（インデックス） |
| 配送手段 | 検索エンジン（Web UI） | CLI + API |
| moat | インデックスの質（PageRank） | ツール知識の質（推薦精度） |
| スケーリング | 自動クローリング | 自動抽出 |

## 2. 現状とビジョンのギャップ

### 現在の実装

```
手動で定義されたツール: 137個（JSON形式）
検索: セマンティック + TF-IDF ハイブリッド
パイプライン: discover → resolve → generate → execute
```

### ビジョンが求めるもの

```
接続すべきツール: 数百万
合成パターン: 事実上無限
対応速度: 新しいツールが生まれた瞬間にキャッチアップ
```

### ギャップの核心

**137 → 数百万。3〜4桁のスケールジャンプが必要。手動キュレーションでは絶対に到達できない。**

つまり、ai-cli の最大の技術課題は:

> **ソフトウェアツールの知識を、自動的に、大規模に、継続的に獲得する仕組み**

---

## 3. フォーカスすべき3つの柱

### 柱1: 自動知識獲得（最重要・最優先）

**問題**: 手動で JSON を書く方式では 137 ツールが限界。
**解法**: パッケージマネージャのメタデータ + ドキュメントから、ツール知識を自動抽出する。

```
npm registry API → パッケージ名、説明、README
  ↓ LLM で解析
  ↓
構造化ツール知識: 何をするか、どう使うか、いつ使うか
```

具体的なソース:

| ソース | メタデータAPI | 抽出可能な情報 |
|---|---|---|
| npm | registry.npmjs.org | name, description, README, bin フィールド |
| PyPI | pypi.org/pypi/{pkg}/json | summary, description, classifiers |
| apt | apt-cache show | Description, Depends |
| Homebrew | formulae.brew.sh/api | desc, homepage, dependencies |
| GitHub | api.github.com | README, releases, topics |

**キーインサイト: LLM を使って LLM の能力を拡張する再帰構造**

ai-cli は LLM を使ってドキュメントを読み、ツール知識を生成する。その知識が別の LLM の能力を拡張する。AI が AI を強くする。

**知識の深さのレベル:**

| レベル | 内容 | 自動化難易度 | 価値 |
|---|---|---|---|
| L1 | 名前 + 説明（何をするか） | 容易 | 基本 |
| L2 | サブコマンド + フラグ（どう使うか） | 中 | 高 |
| L3 | 合成パターン（他のツールとどう組み合わせるか） | 難 | 最高 |

**戦略: L1 を数百万規模で自動生成し、L2-L3 を人気ツールから段階的に深化させる。**

### 柱2: 合成知識（Unix哲学の完成）

単体ツールの知識だけでは不十分。**ツール同士をどう組み合わせるか**の知識が、能力を指数的に拡張する。

```bash
# これらの合成パターンを「知っている」ことが価値
curl API | jq '.data[]' | xargs -I{} command {}
find . -name "*.log" | xargs grep "ERROR" | sort | uniq -c | sort -rn
docker ps -q | xargs docker stop
git log --oneline | head -20 | fzf | awk '{print $1}' | xargs git show
```

合成知識の構造:

```json
{
  "pattern": "API → 構造化データ → バッチ処理",
  "tools": ["curl", "jq", "xargs"],
  "template": "curl {url} | jq '{filter}' | xargs -I{} {command} {}",
  "use_when": "APIからデータを取得し、各項目に対して操作を実行したいとき"
}
```

**獲得方法:**
- GitHub のシェルスクリプトから頻出パターンを抽出
- Stack Overflow の回答からパイプライン例を収集
- LLM に「この2つのツールはどう組み合わせられるか」を生成させる

### 柱3: 効率的な知識配送

数百万のツール知識をどう配送するか。全てをコンテキストに入れることは不可能。

**設計原則: 必要な知識を、必要なときに、最小トークンで届ける**

これは v3 の「トークン効率」の本質を引き継ぐ（数字には囚われないが、設計原則として重要）。

```
全ツール知識をコンテキストに入れる: 不可能（数百万ツール）
MCP 方式（ツール定義をロード）: 重い（3サーバーで 72% 消費）
ai-cli 方式: 意図に対して上位3ツールの知識だけ注入 → 軽い
```

**アーキテクチャ:**

```
ユーザーの意図
  ↓
[検索層] セマンティック検索で候補を絞り込み（ミリ秒）
  ↓
[知識注入] 上位 N 件のツール知識だけを LLM コンテキストに注入
  ↓
[生成層] LLM がコマンド（または合成パイプライン）を生成
  ↓
[実行層] bash で実行
```

この「検索 → 注入 → 生成」のパターンは **Tool RAG** と呼べる。
RAG がドキュメント知識で LLM を拡張するように、Tool RAG がツール知識で LLM を拡張する。

### Static Index（事前構築インデックス）

- パッケージマネージャから自動抽出した数百万のツール知識
- 組み込み: 人気上位 10,000 ツール（シングルバイナリに同梱）
- 拡張: 残りはオンデマンドでダウンロード

### Dynamic Discovery（動的発見）

- Static Index にないツールを実行時に発見
- `npm search`, `apt search`, `pip search` + README 解析
- 「知らないツールに出会ったら、その場で学ぶ」能力

### Community Layer（コミュニティ）

- OSS でツール定義を共同編集
- 自動生成された知識を人間が検証・修正
- 合成パターンの共有

## 5. やらないこと（フォーカスの裏側）

| やらないこと | 理由 |
|---|---|
| AI モデルの開発 | 既存 LLM を使う。モデルは ai-cli の仕事ではない |
| パッケージマネージャの開発 | 既存のもの（npm, apt, pip）を使う |
| MCP サーバーレジストリ | Anthropic の領域（Tool Search） |
| 新しいシェルの開発 | bash を使う。Unix との共存原則 |
| GUI / IDE 統合 | 配送手段は後回し。まず知識の質と量 |
| claude-bridge の拡張 | v3 で凍結済み。知識層が優先 |
| auto-provision の拡張 | 派生機能。知識層が優先 |

**最も重要な「やらない」判断: 手動でツール定義を書き続けること。**

137 → 200 → 300 と手動で増やしても、数百万には永遠に届かない。自動知識獲得の仕組みに投資すべき。

## 6. フェーズ計画

### Phase 1: 自動知識獲得の基盤（最優先）

**目標**: 手動 137 ツール → 自動 10,000+ ツール

- npm registry API からパッケージメタデータを自動取得
- LLM でメタデータ + README → 構造化ツール知識を生成
- 品質検証パイプライン（生成された知識の精度を測定）
- 現在の手動 JSON と同じスキーマに出力（既存エンジンと互換）

**成功指標**: 自動生成されたツール知識で、手動定義と同等以上の検索精度

### Phase 2: 合成知識と検索強化

**目標**: 単体ツール → 合成パターンの知識

- GitHub/StackOverflow からパイプラインパターンを収集・解析
- 合成パターンのインデックス構築
- 検索を拡張: 「〜したい」→ 単体ツール OR 合成パイプラインを提案
- 知識規模: 10,000+ ツール + 1,000+ 合成パターン

**成功指標**: 合成パイプラインの提案が、単体ツール提案より良い結果を出すケースを実証

### Phase 3: スケールと配送

**目標**: 10,000 → 100,000+ ツール、複数の配送チャネル

- PyPI, apt, Homebrew, Cargo からも自動取得
- Dynamic Discovery の実装（Static Index にないツールの実行時発見）
- API/SDK としての公開（他の AI エージェントが利用可能に）
- コミュニティ貢献の仕組み

**成功指標**: 任意のタスクに対して、人間の専門家と同等のツール推薦ができる

### Phase 4: 知識のネットワーク効果

**目標**: 使われるほど賢くなる仕組み

- 利用データからの学習（どのツールが実際に使われたか）
- 失敗からの学習（推薦したツールが不適切だったケース）
- コミュニティが成長の源泉になる

## 7. 設計上の判断ポイント

### Single Binary 制約との両立

数百万のツール知識をシングルバイナリに同梱するのは非現実的。

**解決策: 階層型アーキテクチャ**

```
バイナリ内蔵: 上位 10,000 ツール（圧縮して数MB）
  → インストール直後から価値を発揮
  → Zero config 原則を維持

オンデマンド: 残りのツール知識
  → 初回検索時にダウンロード + ローカルキャッシュ
  → オフラインでも内蔵分は動作
```

### オープンソース戦略

ツール知識ベースをオープンにすることで:
- コミュニティが知識を拡張・修正できる
- ツール開発者自身がai-cli用の定義を提供する動機が生まれる
- 知識の質がクラウドソーシングで向上する

**これは npm が package.json を標準化したのと同じ戦略。ツール開発者が自分のツールの ai-cli 定義を書く世界。**

### 知識フォーマットの進化

現在の JSON スキーマは良い出発点だが、合成パターンや動的発見を含めるには拡張が必要。

```json
{
  "id": "jq",
  "capabilities": ["json-parse", "json-transform", "json-query"],
  "composes_well_with": ["curl", "xargs", "parallel"],
  "composition_patterns": [
    {
      "with": "curl",
      "pattern": "curl {url} | jq '{filter}'",
      "use_when": "API レスポンスから特定のフィールドを抽出"
    }
  ],
  "better_than": [
    { "tool": "python -c 'import json...'", "when": "シンプルなJSON操作", "why": "1行 vs 10行" }
  ]
}
```

## 8. 本質の要約

**ai-cli がフォーカスすべきことは1つ:**

> **ソフトウェアツールの知識を、自動的に、大規模に、継続的に獲得し、AI に届ける仕組みを作ること。**

137 個の手動定義を 200 個にする作業ではない。
自動的に 10,000 → 100,000 → 数百万に拡大する**仕組み**を作ること。

ツールは配送手段。知識がプロダクト。仕組みが moat。
